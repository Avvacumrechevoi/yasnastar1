# Server-driven архитектура Ясны

Этот документ фиксирует целевую архитектуру экрана `/star` после очистки интерфейса и выноса предметной логики на backend. Ключевой принцип здесь простой: **фронтенд ничего не трактует**. Он не знает, какая Ясна что означает, какие механики надо включать, какие полочки подсвечивать, как строится синтез и какой текст показывать в инспекторе. Всё это вычисляет серверный домен Ясны, а клиент получает уже готовое состояние для рендера.

## Главная цель

Система должна состоять из трёх уровней. На первом уровне хранится предметная модель Ясны в БД. На втором уровне работают backend-сервисы, которые собирают модель, рассчитывают геометрию и смысловое состояние звезды, механик, инспектора и синтеза. На третьем уровне фронтенд только отображает готовый ответ и отправляет пользовательские действия обратно в API.

| Слой | Ответственность | Что в нём не должно быть |
| --- | --- | --- |
| База данных | Хранение Ясн, полочек, механик, связей, линз, пресетов и текстов интерпретации | Бизнес-логики вычисления активного состояния |
| Backend-домен | Расчёт состояния звезды, активных механик, инспектора, синтеза и прикладных интерпретаций | JSX, визуальных решений и фронтенд-зависимых эвристик |
| Frontend | Отрисовка звезды, контролов, списков, инспектора и переходов | Хардкода предметных смыслов, локальных таблиц механик и ручной трактовки Ясн |

## Что нужно убрать из UI

Из интерфейса нужно убрать служебные блоки, которые рассказывают пользователю о внутреннем устройстве сервера. Блок вида `BACKEND-DRIVEN РЕЖИМ`, технические подписи вроде «серверный фокус» и пояснения о том, что именно вычислил backend, не несут продуктовой ценности для конечного пользователя. Пользователь должен видеть только содержательные элементы: звезду, выбор Ясны, полезный инспектор, сценарии анализа, синтез при необходимости и рекомендации по работе с ситуацией.

Иными словами, **server-driven** должно быть свойством архитектуры, а не видимым элементом интерфейса.

## Целевая доменная модель в БД

Сейчас предметная модель живёт в каталоге кода. Для правильной архитектуры её нужно перевести в БД и разложить на сущности. Тогда Ясны можно будет редактировать, расширять, версионировать и связывать без переписывания фронтенда.

| Таблица | Назначение | Ключевые поля |
| --- | --- | --- |
| `yasnaModels` | Основная сущность Ясны | `id`, `slug`, `title`, `family`, `summary`, `description`, `isActive`, `version` |
| `yasnaPoints` | Полочки конкретной Ясны | `id`, `yasnaModelId`, `pointIndex`, `label`, `essence`, `rawText`, `tooltip` |
| `yasnaMechanics` | Справочник механик звезды | `id`, `code`, `title`, `shortTitle`, `kind`, `description`, `stroke`, `sortOrder` |
| `yasnaMechanicPoints` | Связь механик с полочками | `id`, `mechanicId`, `pointIndex`, `role` |
| `yasnaMechanicGroups` | Логические группы механик | `id`, `code`, `title`, `description`, `sortOrder` |
| `yasnaMechanicGroupItems` | Связь механик с группами | `id`, `groupId`, `mechanicId`, `sortOrder` |
| `yasnaLessons` | Уроки и методические материалы Ясны | `id`, `yasnaModelId`, `title`, `content`, `sortOrder` |
| `yasnaLenses` | Прикладные линзы анализа для конкретной Ясны | `id`, `yasnaModelId`, `code`, `title`, `summary`, `question` |
| `yasnaPresets` | Сценарные пресеты внутри линзы | `id`, `lensId`, `code`, `title`, `summary`, `question`, `sortOrder` |
| `yasnaPresetMechanics` | Рекомендуемые механики пресета | `id`, `presetId`, `mechanicId`, `sortOrder` |
| `yasnaPresetPoints` | Фокусные полочки пресета | `id`, `presetId`, `pointIndex`, `sortOrder` |
| `yasnaInterpretations` | Тексты интерпретации по Ясне, полочке, механике и режиму | `id`, `yasnaModelId`, `scopeType`, `scopeRef`, `analysisMode`, `title`, `body`, `tone` |
| `yasnaSynthesisRules` | Правила мостов между Яснами | `id`, `primaryYasnaId`, `secondaryYasnaId`, `primaryPointIndex`, `secondaryPointIndex`, `title`, `description`, `riskText`, `sortOrder` |

## Почему именно такая схема

Такая схема отделяет **структуру знания** от **вычисления состояния**. База отвечает на вопрос, что существует в предметной области: какие Ясны, какие у них полочки, какие механики, какие линзы и тексты. Сервисный слой отвечает на вопрос, что из этого активно прямо сейчас для конкретного пользовательского выбора.

Это даёт несколько преимуществ.

Во-первых, можно добавлять новые Ясны и механики без переписывания фронтенда. Во-вторых, можно менять тексты трактовок и сценарные пресеты через данные, а не через условные ветки в React. В-третьих, появляется нормальная тестируемая граница между хранением, расчётом и визуализацией.

## Backend-модули

Целевой backend стоит разложить не в один большой `service.ts`, а на несколько модулей с понятной ответственностью.

| Модуль | Ответственность | Что отдаёт |
| --- | --- | --- |
| `server/yasna/repository.ts` | Чтение Ясн, полочек, механик, линз, пресетов и правил синтеза из БД | Нормализованные доменные записи |
| `server/yasna/catalog-loader.ts` | Временный сидер или адаптер для первичного наполнения БД из существующего каталога | Данные для инициализации |
| `server/yasna/star-engine.ts` | Чистая логика геометрии звезды и вычисления активных механик | Активные связи, подсветки, выбранные и противоположные полочки |
| `server/yasna/inspector-service.ts` | Сборка содержимого инспектора по текущему состоянию | Заголовок, основной текст, контекст, карточки и рекомендации |
| `server/yasna/synthesis-service.ts` | Сопоставление двух Ясн и построение мостов | `bridges`, `risks`, `summary` |
| `server/yasna/query-service.ts` | Оркестрация всего ответа для фронтенда | Полный `YasnaComputedState` |
| `server/yasna/seed.ts` | Первичная загрузка моделей, полочек и механик в БД | Команда инициализации |

## Серверное вычислительное ядро

Центральная идея архитектуры заключается в том, что вычисление состояния звезды должно быть **чистой функцией** поверх доменной модели и пользовательского ввода. Входом служат `yasnaId`, `selectedPoint`, `activeMechanicIds`, `presetId`, `analysisMode` и `secondaryYasnaId`. Выходом становится не кусок геометрии и не набор локальных флагов, а полностью собранное состояние экрана.

| Вход | Где используется | Результат |
| --- | --- | --- |
| `yasnaId` | Загрузка модели Ясны и связанных данных | Активная Ясна |
| `selectedPoint` | Движок звезды и инспектор | Активная полочка, противоположная полочка, подсветки |
| `activeMechanicIds` | Движок звезды | Явно выбранные механики пользователя |
| `presetId` | Пресетный слой | Подбор рекомендованных механик и полочек |
| `analysisMode` | Оркестрация сценария чтения | Свободное чтение, пошаговый анализ или синтез |
| `secondaryYasnaId` | Сервис синтеза | Межъясовые мосты и риски |

## API-контракт для фронтенда

Фронтенду нужен не технический, а продуктовый контракт. Он должен получать уже готовые блоки для отображения, не вычисляя ничего предметного самостоятельно.

| Процедура | Назначение | Ответ |
| --- | --- | --- |
| `yasna.catalog` | Короткий каталог доступных Ясн для селекторов и стартового экрана | Список `id`, `slug`, `title`, `family`, `summary` |
| `yasna.computeState` | Полное server-driven состояние страницы `/star` | `activeYasna`, `availableMechanics`, `activeMechanics`, `inspector`, `negotiation`, `synthesis`, `ui` |
| `yasna.getModel` | Детальная модель конкретной Ясны для админского редактирования или отладки | Полная структура Ясны |
| `yasna.admin.upsertModel` | Создание или обновление Ясны | Сохранённая модель |
| `yasna.admin.seed` | Первичное заполнение БД из существующего каталога | Отчёт о загрузке |

Для пользовательского UI достаточно двух процедур: `catalog` и `computeState`. Все остальные нужны для поддержки, администрирования и роста продукта.

## Как должен работать экран `/star`

Когда пользователь открывает страницу, фронтенд вызывает `yasna.catalog` для списка Ясн и затем `yasna.computeState` для текущей активной Ясны. Дальше любое действие пользователя — выбор полочки, механики, пресета или второй Ясны — не запускает локальную бизнес-логику. Фронтенд просто меняет параметры запроса и повторно получает готовое вычисленное состояние.

Именно так интерфейс остаётся чистым. React отвечает только за визуальное состояние: что раскрыто, что анимируется, какой таб активен визуально, как выглядит выбранная карточка. Но React не решает, какая полочка к чему ведёт и какой смысл имеет механика. Это решает backend.

## Что ещё важно добавить сверх уже обсуждённого

Пользователь справедливо указал на ещё не учтённые элементы. Их нужно заложить сразу.

| Что нужно учесть | Почему это важно | Где реализуется |
| --- | --- | --- |
| Версионирование моделей Ясны | Чтобы менять структуру без потери совместимости | `yasnaModels.version` и отдельные сиды |
| Админский путь загрузки данных | Чтобы база не зависела от ручного SQL | `server/yasna/seed.ts` и admin-процедуры |
| Разделение read model и engine | Чтобы тестировать расчёты отдельно от БД | `repository.ts` + `star-engine.ts` |
| Явная схема интерпретаций | Чтобы тексты не были захардкожены в JSX и не размазывались по сервисам | `yasnaInterpretations` |
| Правила синтеза как данные | Чтобы не кодировать мосты между Яснами вручную во фронте | `yasnaSynthesisRules` |
| Тесты на доменный слой | Чтобы логика звезды не ломалась при развитии каталога | Vitest для engine и query service |
| Отдельный пользовательский и админский API | Чтобы frontend-пользовательский контракт оставался компактным | `yasnaRouter` и `yasnaAdminRouter` |

## Переходный план реализации

Так как система уже работает на каталоге в коде, миграция должна идти поэтапно.

| Этап | Что делаем | Результат |
| --- | --- | --- |
| 1 | Убираем из UI служебные backend-блоки | Чистый пользовательский интерфейс |
| 2 | Нормализуем схему БД для Ясн и механик | Появляется постоянное хранилище |
| 3 | Пишем сидер из текущего `catalog.ts` | Текущие данные переезжают в БД без ручного ввода |
| 4 | Выделяем `repository.ts` и `query-service.ts` | Сервисный слой перестаёт зависеть от in-memory каталога |
| 5 | Переносим `computeState` на чтение из БД | Фронт получает те же ответы уже из серверного хранилища |
| 6 | Добавляем админские процедуры и тесты | Архитектура становится расширяемой и поддерживаемой |

## Итоговый принцип

Итоговая система должна работать так: **данные о Яснах хранятся в БД, сервер собирает из них доменную модель, вычисляет логику звезды и возвращает фронтенду готовое состояние экрана**. Фронтенд остаётся чистым, красивым и лёгким, потому что он не содержит предметной логики, а только качественно отображает серверный результат.

Это и есть правильная архитектура для вашего случая: не «богатый фронтенд с красивыми хардкодами», а **чистый UI + сильный backend-домен Ясны**.
