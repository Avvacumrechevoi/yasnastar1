# Backend-дизайн сервиса Ясны

## Цель

Перенести критичную логику экрана `/star` с клиента на серверный слой так, чтобы фронтенд отвечал только за визуализацию, локальные UI-состояния и отправку пользовательского выбора, а все семантические вычисления Ясны, каталог механик, состав точек, тексты инспектора и правила связей формировались в backend-сервисе.

## Архитектурное решение

Сервис Ясны будет построен как **доменный модуль на сервере**, а не как набор вычислений внутри React-страницы. Для этого данные и правила разделяются на три уровня.

| Уровень | Назначение | Что хранит |
| --- | --- | --- |
| `server/yasna/catalog.ts` | Канонический серверный каталог | Ясны, механики, группы механик, базовые определения точек |
| `server/yasna/service.ts` | Доменная логика | Нормализация полочек, вычисление активных механик, инспектора, подсветок и связей |
| `server/routers/yasna.ts` | API-контракт | tRPC-процедуры для получения каталога и вычисленного состояния звезды |

## Почему не база данных на первом шаге

Текущая задача пользователя состоит не в пользовательском редактировании Ясен, а в переносе механизма с фронтенда на backend. Поэтому на первом этапе источником истины станет серверный каталог в коде. Это даст типизированную и тестируемую доменную модель без лишнего усложнения административными таблицами и CRUD-интерфейсом. Если позже появится задача редактировать Ясны через админку, каталог можно будет вынести в таблицы без смены внешнего API, поскольку фронтенд будет уже зависеть от backend-контракта, а не от локального JSON.

## Доменная модель

| Сущность | Ключевые поля | Роль |
| --- | --- | --- |
| `YasnaDefinition` | `id`, `family`, `title`, `summary`, `lessonCount`, `points`, `lessons`, `notes`, `mechanics` | Каноническое описание конкретной Ясны |
| `MechanicDefinition` | `id`, `title`, `shortTitle`, `alias`, `category`, `kind`, `points`, `stroke`, `fill`, `glow`, `description` | Серверное описание механики и её геометрии |
| `YasnaPointView` | `index`, `rawText`, `label`, `essence`, `tooltip` | Нормализованное представление полочки для UI |
| `YasnaInspectorState` | `title`, `subtitle`, `selectedPoint`, `mechanic`, `oppositePoint` | Серверно вычисленная правая панель |
| `YasnaComputedState` | `yasna`, `mechanics`, `selectedPoint`, `highlightedPoints`, `selectedPointMechanics`, `inspector` | Главный снимок состояния звезды для клиента |

## API-контракт

На первом этапе достаточно двух процедур.

| Процедура | Назначение | Вход |
| --- | --- | --- |
| `yasna.catalog` | Возвращает список Ясен, групп механик и серверный каталог | без входа |
| `yasna.computeState` | Возвращает вычисленное состояние звезды и инспектора | `yasnaId`, `selectedPoint`, `activeMechanicIds` |

### Ответ `yasna.catalog`

Процедура отдаёт только те данные, которые нужны для инициализации интерфейса: список Ясен, механики, группы механик и стартовый `defaultYasnaId`. Это позволит загружать каталог один раз и не держать жёсткую копию в клиентском коде.

### Ответ `yasna.computeState`

Процедура возвращает уже рассчитанные сервером поля:

| Поле | Смысл |
| --- | --- |
| `activeYasna` | Текущая Ясна с нормализованными полочками |
| `activeMechanics` | Подробные объекты активных механик |
| `latestMechanic` | Последняя включённая механика |
| `selectedPointView` | Нормализованная выбранная полочка |
| `oppositePointView` | Противоположная точка для режима противопоставления |
| `selectedPointMechanics` | Активные механики, проходящие через выбранную точку |
| `highlightedPointIndices` | Множество подсвеченных точек |
| `inspector` | Готовые заголовки и описания правой панели |

## Перенос логики из Star.tsx

Ниже перечислены вычисления, которые должны уйти на сервер полностью.

| Текущая логика на клиенте | Новое место |
| --- | --- |

## Что остаётся на фронтенде

На клиенте остаются только интерфейсные состояния и визуальная геометрия, которые завязаны на рендеринг, а не на доменный смысл.

| Остаётся в React | Причина |
| --- | --- |
| положение плавающего окна, drag/resize | это чистый UI-state |
| SVG-координаты точек и подписи | это слой визуализации, а не семантики |
| локальное переключение активного таба до отправки в API | это управление интерфейсом |
| отрисовка дуг, линий, крестообразных фигур | фронтенд должен рисовать, но не интерпретировать смысл |

## Тестовая стратегия

Новый серверный модуль должен покрываться тестами на уровне доменной логики и API.

| Объект тестирования | Что проверяем |
| --- | --- |
| `computeYasnaState` | корректный выбор Ясны по id, fallback на default, подсветка точек, противоположная точка |
| текстовые нормализаторы | извлечение `label`, `essence`, `tooltip` из сырого текста полочки |
| инспектор | корректный текст без выбранной точки, с выбранной точкой и с механикой `contrast` |
| tRPC `yasna.catalog` и `yasna.computeState` | контракт ответа и работа через `appRouter.createCaller` |

## План реализации

Сначала нужно вынести данные и типы из `Star.tsx` в серверный каталог и сервис. Затем следует подключить новый `yasna`-роутер к `appRouter`, написать тесты и только после этого перевести страницу `/star` на `trpc.yasna.catalog` и `trpc.yasna.computeState`.
