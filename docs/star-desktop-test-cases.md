# Формальные test-cases для экрана `/star`

Экран `/star` проверяется как **desktop-инструмент** с тремя независимыми зонами: левой колонкой механик, центральной сценой звезды и правой колонкой контекста. Цель набора тестов — не только обнаружить визуальные дефекты, но и зафиксировать понятные критерии приемки для каждого плотного состояния интерфейса.

| ID | Сценарий | Предусловия | Шаги | Критерий прохождения |
| --- | --- | --- | --- | --- |
| STAR-DESK-01 | Базовая загрузка сцены | Открыта страница `/star` на desktop | Дождаться полной загрузки страницы | Левый столбец, центральная звезда и правый столбец видимы одновременно; экран не разваливается по высоте; нет горизонтального скролла окна |
| STAR-DESK-02 | Компактность списка механик | Открыта страница `/star` | Визуально проверить левый список механик | В списке показаны только короткие названия кнопок; элементы не содержат длинних описаний; плотность списка позволяет видеть значительную часть набора без лишней высоты |
| STAR-DESK-03 | Компактность блока выбора Ясны | Открыта страница `/star` | Проверить верхний блок выбора Ясны | Кнопка открытия библиотеки видна сразу; выпадающий список не содержит длинных пояснений; строка выбора не съедает лишнюю высоту над сценой |
| STAR-DESK-04 | Вмещаемость подписей вокруг звезды | Открыта страница `/star` с заполненной Ясной | Проверить все 12 подписей вокруг точек | Ни одна подпись не обрезана по краям центральной карточки; текст не наезжает на соседние подписи, точки и направляющие |
| STAR-DESK-05 | Нумерация точек внутри кружков | Открыта страница `/star` | Проверить все 12 точек | Номера точек находятся внутри кружков, центрированы и читаются без конфликта с подписями |
| STAR-DESK-06 | Переключение режима «Все» | Открыта страница `/star` | Включить режим `Все` | Механики включаются массово; центральная сцена не теряет читаемость; линии и фигуры видимы как дополнительные слои, а не как хаотичный шум |
| STAR-DESK-07 | Выбор полочки в насыщенном состоянии | Включён режим `Все` | Кликнуть по любой точке звезды | Выбранная полочка получает заметный active-state; правая колонка показывает актуальный контекст выбранной точки; центр не съезжает |
| STAR-DESK-08 | Изолированная прокрутка левой колонки | Контента в левой колонке достаточно для переполнения | Прокрутить только левый столбец | Скролл происходит только внутри левой колонки; центральная сцена и правый столбец не смещаются |
| STAR-DESK-09 | Изолированная прокрутка правой колонки | Включён режим `Все` и выбрана точка | Прокрутить только правый столбец | Скролл происходит только внутри правой колонки; левая колонка и центральная сцена остаются стабильными |
| STAR-DESK-10 | Устойчивость центральной сцены | Есть длинный контент в боковых колонках | Поочерёдно скроллить левую и правую колонки | Звезда остаётся в своей колонке и не подпрыгивает по вертикали из-за соседних панелей |
| STAR-DESK-11 | Правый столбец как единственный инспектор | Открыта страница `/star` | Выбрать точку и несколько механик | Вся детализация отображается только справа; не появляется дублирующее плавающее окно или лишняя нижняя карточка |
| STAR-DESK-12 | Читабельность линий механик | Включить `Единства`, `Борьбы`, `Напротив` | Сравнить центральную сцену до и после включения | Линии читаются именно как линии, а не как едва заметное изменение точек |
| STAR-DESK-13 | Безопасные границы viewport | Открыта страница `/star` на типовом desktop viewport | Проверить верхние, боковые и нижние подписи | Ни одна подпись или кнопка не выходит за видимую безопасную область карточки центральной сцены |
| STAR-DESK-14 | Устойчивость на уменьшенном desktop-окне | Открыта страница `/star` | Уменьшить ширину окна до состояния, близкого к пользовательскому скриншоту | Звезда остаётся читаемой, подписи не разрушают компоновку, а критические элементы не теряют доступность |

## Правила фиксации дефектов

Каждый найденный дефект при прогоне test-cases должен описываться через три поля: **где проявился**, **что именно переполнилось или наложилось**, **какой критерий из таблицы нарушен**. Такой формат нужен, чтобы быстро переводить визуальные наблюдения в конкретные правки layout и затем повторно валидировать их тем же набором сценариев.
