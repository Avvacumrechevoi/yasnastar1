# Архитектура Ясны после переноса логики на backend

Система перестроена в **server-driven** модель. Это означает, что фронтенд больше не хранит предметную логику Ясны, не вычисляет переговорные состояния и не решает, какие механики, связи, подсказки и режимы надо показать. Теперь фронтенд отвечает только за **отображение**, пользовательские действия и навигацию, а весь смысловой слой находится на сервере.

## Что изменилось по сути

Раньше важная часть модели жила рядом с интерфейсом: каталог Ясн, механики, режимы анализа и часть вычислений были слишком близко к экрану `/star`. Это делало интерфейс тяжелее, усложняло развитие и смешивало визуальный слой с доменной логикой. Теперь эта связка разнесена по слоям.

| Слой | Роль | Что в нём находится |
|---|---|---|
| **База данных** | Источник предметных данных | Ясны, полочки, механики, связи, пресеты, переговорные интерпретации |
| **Репозиторий** | Чтение и сборка доменных снимков | Загрузка данных из БД, нормализация, fallback-наполнение |
| **Runtime-каталог** | Актуальный серверный снимок | Уже собранный каталог Ясн для быстрых вычислений сервиса |
| **Доменный сервис Ясны** | Вычислительное ядро | Состояние звезды, активные механики, инспектор, синтез, режимы анализа |
| **tRPC API** | Контракт между UI и backend | `catalog`, `computeState` |
| **Фронтенд** | Чистый UI-слой | Отрисовка звезды, вкладок, overlay, реакция на клики |

## Как теперь открывается страница `/star`

При открытии страницы фронтенд запрашивает у сервера каталог Ясн. Перед отдачей ответа сервер обновляет runtime-каталог из репозитория. Репозиторий, в свою очередь, читает данные из БД и собирает нормализованный снимок домена. Уже этот снимок используется сервисом Ясны.

После этого фронтенд получает не «кусочки логики», а готовые данные для интерфейса: список Ясн, механик, групп механик и режимов анализа. Когда пользователь выбирает Ясну, полочку, механику или режим, фронтенд отправляет только параметры выбора, а сервер возвращает уже **вычисленное состояние** экрана.

## Что именно считает backend

Backend теперь отвечает за все предметные решения, которые раньше могли разрастаться во фронтенде.

| Подсистема | Что считает сервер |
|---|---|
| **Выбор Ясны** | Какая Ясна активна, какие полочки и интерпретации у неё доступны |
| **Механики** | Какие механики активны, какие точки и связи надо подсветить |
| **Инспектор** | Какой заголовок, описание, контекст и пояснение показать справа |
| **Переговорные режимы** | Свободный режим, guided-режим, синтез и связанные рекомендации |
| **Пресеты** | Какие механики и подсказки активировать для конкретного сценария |
| **Синтез** | Мосты, риски и рекомендации между двумя Яснами |

Это важно, потому что теперь любое развитие метода Ясны можно делать в одном месте — в backend-домене, без разрастания условной логики в React-компонентах.

## Почему такая архитектура правильнее

Главное преимущество — разделение ответственности. Фронтенд больше не знает, **почему** именно такая точка активна, почему эта механика влияет на такие полочки и почему именно этот текст показывается в инспекторе. Он знает только, **что нужно показать**.

Это даёт несколько практических выгод.

| Выгода | Что это даёт проекту |
|---|---|
| **Чистый UI** | Компоненты проще читать, поддерживать и переиспользовать |
| **Единая логика** | Нет расхождения между разными экранами и сценариями |
| **Расширяемость** | Новые Ясны и механики можно добавлять через БД и backend-слой |
| **Тестируемость** | Логику можно проверять витестами без браузерной сцены |
| **Контролируемая эволюция** | Можно развивать переговорные режимы и синтез, не ломая UI |

## Что реализовано в этой итерации

В этой версии были сделаны четыре ключевые вещи.

Во-первых, со страницы `/star` убран служебный технический блок, который объяснял backend-driven режим прямо пользователю. Интерфейс снова стал продуктовым и визуально чище.

Во-вторых, схема Ясны переведена на модель с БД и репозиторием. Это означает, что доменные сущности больше не завязаны только на in-memory каталог.

В-третьих, сервис Ясны переведён на runtime-каталог, который обновляется через серверный слой и используется как единый источник для `catalog` и `computeState`.

В-четвёртых, добавлены тесты не только на вычислительное ядро, но и на загрузку server-driven каталога и его использование в API.

## Как это будет развиваться дальше

Теперь добавление новой Ясны или новой механики должно происходить не через фронтенд, а через доменную модель и серверное хранилище. Практически это выглядит так: сначала описывается новая сущность в БД, затем репозиторий учится её читать, после этого сервис включает её в вычисления, и только в самом конце фронтенд автоматически отображает уже пришедшие с сервера данные.

То есть дальнейший путь такой:

| Шаг | Где делается изменение |
|---|---|
| Добавить новую Ясну | БД + репозиторий |
| Добавить новую механику | БД + доменный сервис |
| Добавить новый режим анализа | backend-сервис + API-контракт |
| Изменить отображение | фронтенд без изменения предметной логики |

## Как это работает для пользователя

Для пользователя всё выглядит просто. Он открывает страницу, видит уже готовую Ясну, выбирает нужную полочку и механики, а интерфейс показывает результат. Но за этим теперь стоит правильная архитектура: фронтенд не «додумывает» бизнес-смысл сам, а получает его из backend-ядра.

Именно это вы и хотели: **красивый чистый фронт**, а вся логика звезды, механик, интерпретаций и серверных данных — **на backend**.
