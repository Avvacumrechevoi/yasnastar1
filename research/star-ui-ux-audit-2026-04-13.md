# Аудит `/star` — UI/UX и сценарии

## Ключевые наблюдения

Текущая страница `client/src/pages/Star.tsx` совмещает в одном файле несколько ролей: локальное состояние выбора Ясны, механик, полочки и режимов анализа; вычисление производных UI-данных для inspector-окна; геометрию звезды и подписей; а также полную разметку левой панели, центральной сцены и правого окна. Это делает правки медленными и рискованными.

Серверный слой уже имеет сильную базу: `server/routers/yasna.ts` тонкий и передаёт вычисление состояния в `server/yasna/service.ts`. Однако сам `service.ts` очень крупный и объединяет доменную модель, конфигурацию переговорных режимов, кеширование, вычисление inspector-состояния, синтез и рекомендации в одном месте.

## Текущая карта пользовательских сценариев

| Сценарий | Текущая точка входа | Замечание |
| --- | --- | --- |
| Выбор Ясны | `selectYasna()` в `Star.tsx` | При смене Ясны сбрасываются полочка, механики и режимы; поведение понятное, но сильно связано с общим файлом страницы. |
| Выбор полочки | `selectPoint()` в `Star.tsx` | Локально меняет только `selectedPoint`, а смысл приходит с сервера через `computeState`. |
| Выбор механики | `toggleMechanic()` в `Star.tsx` | UI-кнопки есть, но логика окна и визуального ответа смешана с лишними inspector-блоками, из-за чего поведение выглядит неоднозначным. |
| Сброс/выбрать все механики | `toggleAllMechanics()` | Семантика смешивает «Все» и «Сброс», нужен более явный UX-паттерн. |
| Контекстное окно | `buildFloatingInspectorSrcDoc()` + производные `inspector*` поля | Сейчас окно перегружено extra-карточками уроков, источников и подсказок, хотя пользователь просит только контекст выбранной Ясны, механики и полочки. |
| Guided / Synthesis режимы | `analysisMode`, `activePresetId`, `secondaryYasnaId` | Логика уже есть, но мешает упрощённому UX главного сценария `/star`. |

## Основные архитектурные проблемы

| Зона | Проблема | Направление рефакторинга |
| --- | --- | --- |
| Клиентская страница | `Star.tsx` слишком монолитен | Разделить на scene state, inspector presenter, mechanics panel, yasna tabs, star geometry presenter |
| Inspector | В окно подмешиваются `extraCards` про уроки, источники и подсказки | Свести модель окна к трём контекстам: активная Ясна, активная механика, выбранная полочка |
| Серверное состояние | `service.ts` перегружен разнородной логикой | Выделить селекторы/билдеры: catalog normalization, interaction state, inspector state, negotiation modes |
| Тесты | Есть сильные серверные тесты, но почти нет явных UI/state test-кейсов страницы | Добавить unit-тесты селекторов состояния и рендер-логики inspector/geometry |
| Геометрия подписей | Исправления уже вносились, но логика layout разбросана по странице | Вынести расчёт label-box и симметрии по секторам в отдельный модуль с тестами |

## Что уже подтверждено по тестам

Файл `server/yasna.service.test.ts` уже покрывает нормализацию label/tooltip, opposite point, guided/synthesis state и router-level вызовы. Это хорошая база, но она почти не проверяет упрощённый UX-контракт контекстного окна и связку «нажал Ясну → нажал механику → нажал полочку».

## Следующий шаг

Следующим шагом нужно собрать точную карту файлов, вынести state-контракт контекстного окна и затем перейти к коду: сначала упростить данные inspector-окна и логику применения механик для каждой Ясны, затем отдельно стабилизировать геометрию подписей у 3 и 9.

## Browser verification update — 2026-04-13

- В превью `/star` сценарий выбора механики уже работает в Ясне «Атмосфера встречи»: после клика по «Опоре» справа показываются только блоки **Ясна**, **Механика** и после клика по полочке — **Полочка**.
- В Ясне «Домашние животные» клик по «Опоре» тоже отрабатывает: на сцене появляется ромб механики, а контекстная панель справа меняется на **Ясна = Домашние животные** и **Механика = Крест Опоры**.
- На текущем превью механизм уже привязан к активной Ясне, но нужно продолжить полный сценарный проход по остальным Яснам и дочистить геометрию крайних боковых подписей.

### Дополнительная live-проверка контекстной панели

В Ясне «Домашние животные» связка `Ясна → механика Опора → полочка Коровник` отрабатывает: ромб механики включается, а справа показываются блоки **Ясна**, **Механика** и **Полочка**. Однако UX-контракт ещё не доведён до конца: внутри блока **Механика** выводится динамическая фраза вида «Механика сейчас включена слева, но выбранная полочка не входит в её контур», тогда как по требованию пользователя здесь должно оставаться только название механики и её краткое базовое описание без лишних сценарных пояснений.

### Подтверждение после hot-reload

После последней правки live-превью показывает правильный контракт блока **Механика**: при выборе «Опоры» справа остаются только заголовок механики и её короткое описание без дополнительной фразы о том, входит ли выбранная полочка в контур механики. Это соответствует требованию пользователя о чистом контекстном окне, завязанном только на выборе Ясны, механики и полочки.

### Проверка Ясны «Домашние животные»

В live-превью после рефакторинга слово **«Коровник»** больше не обрезается по левому краю сцены. Переключение на Ясну **«Домашние животные»** сохраняет чистую правую панель: блок Ясны остаётся основным, а до выбора механики и полочки не появляется лишних вложенных карточек.

### Проверка механик по отдельной Ясне

В live-превью для Ясны **«Домашние животные»** клик по механике **«Опора»** отрабатывает корректно: справа появляется только отдельный блок механики с её кратким описанием, без лишних служебных или сценарных карточек. Это подтверждает, что механики теперь применяются в контексте конкретной выбранной Ясны, а не только глобально.
